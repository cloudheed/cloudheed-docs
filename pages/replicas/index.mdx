# Read Replicas

Scale read operations with PostgreSQL read replicas.

import { Callout } from 'nextra/components'

## Overview

Read replicas are read-only copies of your primary database that:

- **Scale reads** - Distribute read queries across replicas
- **Reduce latency** - Place replicas closer to users
- **Improve availability** - Automatic failover when enabled

<Callout type="info">
  Read replicas are available on Large tier and above.
</Callout>

## Create Read Replica

<span className="method-badge method-post">POST</span> `/v1/databases/{id}/replicas`

```bash
curl -X POST https://api.cloudheed.com/v1/databases/db-abc123/replicas \
  -H "Authorization: Bearer YOUR_TOKEN" \
  -H "Content-Type: application/json" \
  -d '{
    "name": "read-replica-us",
    "region": "ash"
  }'
```

### Parameters

| Field | Type | Required | Description |
|-------|------|----------|-------------|
| `name` | string | Yes | Replica name |
| `region` | string | No | Region (default: same as primary) |

### Response

```json
{
  "id": "replica-123",
  "name": "read-replica-us",
  "database_id": "db-abc123",
  "region": "ash",
  "status": "provisioning",
  "host": "replica-123.db.cloudheed.cloud",
  "port": 5432,
  "replication_lag_bytes": 0,
  "created_at": "2026-03-17T10:00:00Z"
}
```

## Promote Replica

Promote a replica to become a standalone primary database.

<span className="method-badge method-post">POST</span> `/v1/databases/{id}/replicas/{replica_id}/promote`

```bash
curl -X POST https://api.cloudheed.com/v1/databases/db-abc123/replicas/replica-123/promote \
  -H "Authorization: Bearer YOUR_TOKEN"
```

<Callout type="warning">
  Promotion is irreversible. The replica becomes a separate database and stops receiving updates from the primary.
</Callout>

## Replication Lag

Monitor replication lag to ensure data consistency:

| Metric | Description |
|--------|-------------|
| `replication_lag_bytes` | Bytes behind primary |
| `replication_lag_seconds` | Estimated time behind |

<Callout type="info">
  Typical replication lag is under 1 second. High lag may indicate network issues or heavy write load.
</Callout>

## Connection Pooling

Use the replica connection string for read queries:

```javascript
const { Pool } = require('pg');

// Primary for writes
const primary = new Pool({
  connectionString: 'postgresql://cloudheed:pass@db-abc123.db.cloudheed.cloud:5432/postgres'
});

// Replica for reads
const replica = new Pool({
  connectionString: 'postgresql://cloudheed:pass@replica-123.db.cloudheed.cloud:5432/postgres'
});

// Write to primary
await primary.query('INSERT INTO users (name) VALUES ($1)', ['Alice']);

// Read from replica
const result = await replica.query('SELECT * FROM users');
```

## Best Practices

1. **Route reads to replicas** - Use connection pooling to separate read/write traffic
2. **Monitor replication lag** - Alert on high lag values
3. **Consider replica location** - Place replicas near your users
4. **Test failover** - Periodically test promoting a replica
5. **Account for eventual consistency** - Reads from replicas may be slightly behind
